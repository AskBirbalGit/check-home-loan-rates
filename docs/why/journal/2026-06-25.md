## Spread-allocation CIBIL rate curve (replaces per-band averaging)

- **Ask:** Stop suggesting rates with the old 50-point band averaging. CIBIL→rate should
  follow a non-linear curve: 800+ profiles sit near the lender's best rate, 775-800 barely
  moves, and the penalty steepens sharply below 750. The user specified the shape as a fixed
  allocation of each lender's *own* rate spread: >800 = 3%, 775-800 = 7%, 750-775 = 10%,
  700-750 = 20%, 600-700 = 60% (cumulative knots 0.03/0.10/0.20/0.40/1.00). Final displayed
  rate must floor DOWN to the lower 0.05 (7.19→7.15, 7.16→7.15, 7.20→7.20), computed raw then
  floored.
  (Slice shape was iterated several times — 20/30, 15/35, 10/15, 20/30/35, 5/5/10/20/60 —
  settling on 3/7/10/20/60 for >800/775-800/750-775/700-750/600-700, putting the bulk of the
  penalty into the 600-700 band; final-rounding rule was briefly nearest-0.05 then changed to
  floor-to-lower-0.05.)
- **Changed:** `lib/rate-engine.ts` — added a `CURVE` knot table + `curveFraction(score)`
  (linear interpolation between knots, score clamped to [600,850], returns a raw fraction).
  Rewrote `rateFor` to read only two endpoints per lender+employment — `R_min` = low of the
  `800+` cell, `R_max` = high of the `<700` cell (`+` stripped) — and return
  `roundToRateStep(R_min + curveFraction(score) * (R_max − R_min))`. `roundToRateStep` floors
  the raw rate down to the lower 0.05 increment (it replaced the old per-band `floorToRateStep`).
  Removed the now-dead `bandFor`,
  `averagedRate`, `LOW_CIBIL_UPLIFT`, `STEPS`, and the derived 600-649 uplift branch. Rewrote
  the file header doc block.
- **Why:** The old engine averaged within each published 50-point band and floored, which made
  the curve too linear and pushed 800-score profiles well above the lender's true floor. The
  spread-allocation curve is purely endpoint-driven: it imposes the user's intended band-wise
  shape on top of each institution's own R_min→R_max spread, so wider-spread lenders spread the
  same shape over more bps while narrow-spread lenders (e.g. SBI 1.45%) stay tightly clustered
  near the floor at the top. Mid-band published values (750-799/700-749) are intentionally
  discarded — only the two anchors define the spread. The final raw rate is floored DOWN to the
  lower 0.05 (conservative — never quotes a rate below what the curve gives), applied only to
  the final raw value so interpolation error never compounds across segments. The separate
  600-649 risk uplift retires because the
  600-700 slice (40% of the spread) now carries the bottom of the curve directly.
- **Verification:** `npx tsc --noEmit` clean; grep confirms no remaining refs to the removed
  symbols. SBI salaried (R_min 7.25, R_max 8.70) → 850/800/775/750/700/600 =
  7.25/7.25/7.35/7.50/7.80/8.70 (floored). Floor unit-checks: 7.19→7.15, 7.16→7.15, 7.20→7.20.
  No lender has a zero/negative/tiny spread (audited all 32 first-party
  rows × sal/se before implementing).
